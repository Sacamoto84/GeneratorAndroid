# Скриптовый движок: приведение в рабочее состояние

Дата: 2026-07-21
Статус: спека к реализации
Основание: ревью `features/script/*` и `screens/scripting/*`

## Проблема

Скриптовый интерпретатор не исполняется: единственный драйвер `hub.script.run()` в
`VMMain4.launchScriptScope()` закомментирован вместе с самим вызовом `launchScriptScope()`.
Кнопка Play меняет состояние, но программа не выполняется.

При включении «как есть» движок нежизнеспособен: разбор строк не валидируется, любое
исключение приводит к вечному циклу на битой строке, поиск `ENDIF` уходит за границы
списка, флаги останова не потокобезопасны, `DELAY` протекает между запусками.

## Цели

1. Скрипт реально исполняется по кнопке Play, останавливается по Stop, ставится на паузу.
2. Ошибка в скрипте останавливает исполнение и печатается в консоль с номером строки.
3. Разбор строки — в одном месте, с валидацией; исполнение работает с типизированной командой.
4. Вложенные `IF/ELSE/ENDIF` работают.
5. Файловые операции не падают на нестандартном содержимом и не выполняются в теле composable.

Не входит в объём: новые команды языка (`COPY`, `TEXT`, `PRINTF`, `YIELD`) — их нет на
экранной клавиатуре, то есть ввести их пользователь всё равно не может. Восстанавливается
только `FM[1 2] BASE`, которая на клавиатуре есть, а в интерпретаторе закомментирована.

## Архитектура

### Новый файл `features/script/ScriptCommand.kt`

Разбор строки в типизированную команду + ошибки как исключения с номером строки.

```
ScriptException(line: Int, message: String)

Operand = Const(value: Float) | Reg(index: Int)

Cmd =
  Nop                                            // "?" или пустая строка
  End
  Else
  EndIf
  Goto(target: Int)
  Delay(ms: Long)
  Load(dst: Int, src: Operand)
  Arith(isPlus: Boolean, dst: Int, src: Operand)
  If(left: Int, op: CompareOp, right: Operand)
  GenSwitch(ch: Int, block: CR|AM|FM, on: Boolean)      // CH1 FM ON
  GenValue(ch: Int, block: CR|AM|FM, param: FR|BASE|DEV, value: Operand)
  GenMod(ch: Int, block: CR|AM|FM, name: String)        // CR1 MOD 01_Sine
```

Правила разбора:

- Разделитель — любой пробельный участок, пустые токены отбрасываются.
- Регистр: префикс `F` или `R` + индекс `0..9`. `R` и `F` адресуют один и тот же массив
  (исторически; целочисленных регистров в движке нет) — принимается ради старых файлов.
- Индекс регистра вне `0..9` → ошибка разбора, а не `ArrayIndexOutOfBounds`.
- Операнд: регистр либо `Float`; не разобралось → ошибка с текстом токена.
- Недостаточно токенов → ошибка «мало аргументов», а не `IndexOutOfBounds`.
- Неизвестная команда → ошибка. Раньше молча пропускалась с `pc++`.
- `FM ... BASE` пишет частоту несущей канала (`ch*_Carrier_Fr`) — как задумано в шапке
  `Script.kt`.

### `Script.kt`: движок владеет своим циклом

Вместо `suspend fun run()`, который должен был кто-то вызывать снаружи:

- `Script` держит `CoroutineScope(SupervisorJob() + Dispatchers.Default)` и `job` текущего
  прогона. Класс — синглтон Hilt, живёт всё приложение.
- `START` → сброс `pc` и регистров, запуск job.
- `PAUSE` → отмена job, `pc` сохраняется.
- `RESUME` → запуск job с текущего `pc`.
- `STOP` / `EDIT` → отмена job, сброс `pc` и регистров.
- Флаги `end`/`endTime` удаляются: останов — это отмена корутины, задержка — `delay(ms)`,
  который сам отменяем и ничего не «протекает» между запусками.
- Публичное поле `end` больше не мутируется из UI (сейчас так делает нижняя панель).
- Отмена корутины асинхронна: остановленный прогон может доработать текущую инструкцию
  и затереть уже сброшенные `pc` и регистры. Поэтому у прогона есть номер поколения
  (`generation`), и записи `pc`, регистров и `state` идут только от актуального прогона.

Цикл прогона:

```
while (isActive) {
    snapshot = list.toList()
    if (pc !in snapshot.indices) -> ошибка "PC вне списка"
    cmd = parseCommand(snapshot[pc])
    execute(cmd, snapshot)          // сам двигает pc
    if (++steps % YIELD_EVERY == 0) delay(1)   // не занимать ядро в петле без DELAY
}
```

Ошибка: печать `Ошибка в строке N: текст` в консоль скрипта, состояние → `ISTOPPING`,
`pc` остаётся на битой строке для диагностики, регистры не чистятся.

`END`: нормальный выход, состояние → `ISTOPPING`, `pc` → 0.

### Переходы с учётом вложенности

Одна функция поиска парной ветки, работающая по снимку списка и ограниченная его длиной
(не `PC_MAX`):

- `IF` с ложным условием → искать с `pc+1` парный `ELSE` (перейти на строку после него)
  или парный `ENDIF` (перейти на него). Вложенные `IF` увеличивают глубину, `ENDIF` —
  уменьшает; на глубине 0 срабатывает совпадение.
- `ELSE`, до которого дошла истинная ветка → искать парный `ENDIF` с той же логикой.
- Парная строка не найдена до конца списка → ошибка «нет ENDIF», а не выход за границы.

`GOTO` проверяет, что цель попадает в `snapshot.indices`.

`PC_MAX` удаляется: список динамический, константа только вредила.

### Отображение хода исполнения

- `DELAY` ждёт, и только потом двигает `pc`: пока идёт задержка, подсвечена сама строка
  `DELAY`, а не следующая. Плата — пауза посреди задержки сбрасывает её отсчёт, при
  возобновлении задержка отрабатывает заново.
- Регистры движок правит на каждой инструкции, поэтому наблюдаемым делать сам массив
  дорого. Рядом живёт `registers: StateFlow<List<Float>>` — снимок для экрана, который
  публикуется не чаще 50 мс и обязательно в конце прогона (END, ошибка, старт, останов).
  Раньше `RegisterViewDraw` получал голый `FloatArray`, и значения на экране не менялись
  вовсе, пока что-нибудь другое не вызовет рекомпозицию.
- Список команд подкручивается к текущей строке, когда она уходит за пределы экрана.

### Логирование

`Script.log()` (пустая заглушка) заменяется на `var logger: (String) -> Unit`, по умолчанию
`Timber`. Слой UI (`VMScripting`) подключает к нему `consoleLog` — модуль `features` не
должен знать про `screens`. Движок пишет из своего потока, а консоль — Compose-состояние,
поэтому `VMScripting` уводит запись в главный поток (так и было задумано в исходном коде,
где `log()` оборачивался в `Dispatchers.Main`).

Логируются: старт, останов, `END`, ошибки. Не логируется каждая инструкция — цикл горячий.

### `ScriptUtils.kt`

- `readScriptFileToList`: пустые строки отбрасываются; ведущий префикс `N:` снимается
  регуляркой `^\s*\d+\s*:`, а не `split(":")[1]` — строка без префикса больше не роняет
  чтение, текст с `:` внутри не обрезается; пустой файл даёт пустой список, а не
  `NoSuchElementException`.
- `filesInScriptToList`: фильтр по расширению `.sk` и возврат имени без расширения —
  сейчас `dropLast(3)` делает UI, вслепую, для любого файла в папке.
- `renameScriptFile`: ловится `Exception`, а не только `IOException`.

### UI

- `ScriptTable`: чтение списка файлов переносится из тела composable в `LaunchedEffect` —
  сейчас дисковый листинг выполняется на каждой рекомпозиции и мутирует состояние прямо
  во время композиции.
- `BottomAppBarScript`: возобновление через `command(RESUME)` вместо ручной установки
  `script.state` и `script.end`.
- `VMMain4`: мёртвый `launchScriptScope()` удаляется — движок сам себя запускает.

## Проверка

- `:app:compileDebugKotlin` — зелёный.
- JVM-тесты: `junit:junit:4.13.2` был закомментирован в `app/build.gradle`, включается.
  `app/src/test/.../ScriptCommandTest.kt` покрывает разбор всех команд, отказы разбора
  (регистр вне диапазона, нехватка аргументов, мусор, неизвестная команда) и поиск парной
  строки с вложенными `IF`. Прогон: `:app:testDebugUnitTest`.
- Ручная проверка на устройстве: Play гоняет скрипт, Pause замирает, Stop сбрасывает,
  битая строка останавливает с сообщением в консоли. Сценарий из шапки `Script.kt`:
  `LOAD F1 1000` / `IF F1 < 10000` / `CR1 FR F1` / `PLUS F1 100` / `DELAY 50` / `GOTO 2` /
  `ENDIF` / `END`.

## Осталось за рамками

- `COPY`, `TEXT`, `PRINTF`, `YIELD` из старой документации не реализованы: их нет на
  экранной клавиатуре, ввести их нельзя. Теперь такая строка даёт понятную ошибку
  вместо молчаливого пропуска.
- Команд для режима FM min/max (`ch*FmMin`, `ch*FmMax`) в языке нет — нужна и работа
  по клавиатуре, и новые команды.
- Редактирование списка во время исполнения по-прежнему возможно из состояния паузы;
  движок берёт снимок списка на каждой итерации, так что падения не будет, но `pc`
  может указать не на ту строку.
