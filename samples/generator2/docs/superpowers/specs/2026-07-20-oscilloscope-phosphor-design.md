# Oscilloscope Phosphor Accumulator — Design

**Дата:** 2026-07-20
**Статус:** утверждён, готов к написанию плана

## Проблема

Осциллограф рисует каждый аудиосэмпл отдельной точкой `GL_POINTS`
(`MyGLRendererOscill.kt`), координата X берётся из `gl_VertexID`. Отсюда три
дефекта.

### 1. Стробоскопический эффект при кратных частотах

При развёртке 256 окно истории равно 589824 float (294912 сэмплов на канал), а
экран шириной ~1440 px. На один пиксельный столбец приходится ~205 сэмплов
канала. Если частота сигнала делит частоту дискретизации нацело (4410 Hz при
44100 Hz — ровно 10 сэмплов на период), сэмплы попадают в одни и те же 10 фаз.
В столбце оказывается 10 различных значений Y вместо плотного заполнения.

Это не алиасинг: сигнал далеко от Найквиста, спектр цел. Это когерентная
выборка. Расстройка частоты на единицы герц делает период нецелым, фаза
плывёт, столбец заполняется — но это случайность, а не работающий механизм.

Настоящий осциллограф не имеет этого дефекта, потому что луч непрерывен, и
яркость точки экрана определяется временем задержки луча, а не количеством
попавших в неё отсчётов.

### 2. Блендинг не включён

`glEnable(GL_BLEND)` отсутствует во всём рендерере. Альфа `0.10001`, заданная в
вершинном шейдере, драйвером игнорируется — точки пишутся непрозрачными.
Накопления плотности нет; виден только последний отрисованный сэмпл столбца.
Эффект свечения сейчас не работает вообще.

### 3. Стоимость передачи данных

`vertexBuffer` — client-side указатель на нативную память. Каждый
`glDrawArrays` заставляет драйвер копировать ~2.3 МБ и рисовать 589824
`GL_POINTS`. При развёртке 256 это повторяется каждый кадр, хотя за кадр
приходит лишь ~735 новых сэмплов на канал.

## Решение

Заменить точечный рендер на фосфорный аккумулятор: C++ растеризует отрезки
между соседними сэмплами в float-сетку с весом, обратным длине отрезка, а
OpenGL выводит сетку как текстуру. Архитектура повторяет уже работающий в
проекте FFT-спектрограф: построчная (постолбцовая) дозапись вместо перегона
всего массива.

### Ядро: `PhosphorGrid.h`

Заголовочный класс на чистом C++, без зависимостей от JNI и OpenGL.

Сетка: `W` столбцов × `512` бинов, хранение **column-major**, по 2 float на
ячейку — канал L и канал R. Столбец лежит в памяти непрерывно, поэтому уходит
в `glTexSubImage2D` без промежуточного копирования. При `W = 1440` сетка
занимает 5.9 МБ.

Растеризация пары соседних сэмплов канала:

```
xa, xb = позиция сэмплов по столбцам (float)
ya, yb = позиция сэмплов по бинам   (float)

steps = max(|xb - xa|, |yb - ya|, 1)
w     = norm / steps

повторить steps раз: splat(x, y, w) с билинейным распределением по 4 соседям
```

Полная энергия отрезка постоянна и не зависит от его длины. Это прямой
интеграл времени задержки луча: крутой фронт растягивает ту же энергию на
много бинов и даёт тусклый след, пологая вершина синуса концентрирует её и
даёт яркий.

Стробоскопический эффект устраняется по построению. При 4410 Hz отрезок между
соседними сэмплами проходит через все промежуточные бины вместо прыжка между
десятью фиксированными точками.

`norm = 1 / (число сегментов на столбец)` — яркость не зависит от выбранной
развёртки.

### Два режима работы

Порог — `compressorCount >= 32`, тот же, что используется сейчас для выбора
размера точки.

**Roll (развёртка ≥ 32).** Истинный кольцевой буфер. Обрабатываются только
новые сэмплы: ~735 за кадр вместо 589824. Указатель `writeCol` движется по
кругу, столбец обнуляется непосредственно перед записью. Uniform `ringOffset`
сдвигает чтение текстуры в шейдере. Затухание не применяется: послесвечение
создаёт сама прокрутка.

За один аудиопакет продвижение составляет `W / itemCount` столбцов — величина
дробная, поэтому позиция накапливается во float-аккумуляторе.

**Sweep (развёртка < 32).** Сетка пересчитывается из всего окна каждый кадр, с
фосфорным затуханием:

```
grid *= DECAY          // константа 0.85, без элемента управления в UI
grid += кадр * norm
```

Установившееся значение равно `norm / (1 - DECAY)`; шейдерный `gain`
компенсирует этот множитель, поэтому яркость одинакова в обоих режимах.

### JNI: `PhosphorJni.cpp`

Использует существующий глобальный экземпляр из `FloatDirectBuffer.cpp:9`:

```cpp
extern AudioHistoryBuffer audioHistoryBuffer;
```

Интерфейс:

| Функция | Назначение |
|---|---|
| `configure(width, bins, compressorCount, layoutMode)` | Любое изменение параметров вызывает `clear()` |
| `update(): IntArray` | Возвращает `[startCol, count]` — грязные столбцы |
| `gridBuffer(): ByteBuffer` | Direct-буфер всей сетки |
| `ringOffset(): Float` | Смещение кольца в диапазоне `[0, 1)` |

### Рендерер

`MyGLRendererOscill` переписывается: вместо `GL_POINTS` по числу сэмплов —
полноэкранный quad из 4 вершин и текстура `RG16F`. Каждый кадр
`glTexSubImage2D` заливает только грязные столбцы.

Фрагментный шейдер:

```glsl
vec2  e = texture(grid, vec2(fract(uv.x + ringOffset), uv.y)).rg;
float l = 1.0 - exp(-e.r * gain);
float r = 1.0 - exp(-e.g * gain);
fragColor = vec4(colL * l * visL + colR * r * visR, 1.0);
```

Экспонента даёт мягкое насыщение: ядро луча выходит на полную яркость, ореол
остаётся плавным.

Видимость каналов (`bools[1]`, `bools[2]`) остаётся в шейдере — переключение
не требует пересчёта сетки. Режим совмещения каналов (`bools[0]`) переносится в
C++, в функцию `yOf()`; его изменение вызывает `clear()`.

Отсутствие `glEnable(GL_BLEND)` перестаёт быть дефектом: накопление происходит
в сетке, а не во фреймбуфере, и блендинг больше не участвует в формировании
свечения.

## Границы работы

В объём входит только осциллограф. Рендерер фигур Лиссажу
(`MyGLRendererLissagu`) остаётся без изменений; при необходимости он получит
отдельную спецификацию.

Элемента управления послесвечением в UI не будет — `DECAY` остаётся константой
в коде.

## Верификация

Хост-компилятора C++ на машине разработки нет (проверены `g++`, `clang++`,
`cl`; NDK собирает только под Android), поэтому юнит-тесты нативного ядра
запустить негде. Автоматических тестов в этой работе не будет по решению
владельца проекта.

Проверка выполняется сборкой и осмотром на устройстве:

1. `.\gradlew.bat :app:assembleDebug` завершается успешно.
2. Синус 4410 Hz при частоте дискретизации 44100 Hz на развёртке 256 даёт
   плотно заполненный столбец без разрежённого узора из отдельных точек.
3. Расстройка сигнала на несколько герц не меняет картину качественно —
   заполнение остаётся тем же.
4. На развёртках 0.125–16 форма сигнала читается, точки сливаются в
   непрерывный след, послесвечение видно при непериодическом сигнале.
5. Оба канала сохраняют свои цвета, переключение видимости и режима совмещения
   работает.
6. Пауза замораживает изображение.
7. Частота кадров при развёртке 256 не ниже, чем до изменений.
