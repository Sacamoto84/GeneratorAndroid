# Редактор графа нод — дизайн

Дата: 2026-07-22

## Задача

Дать второй способ управлять генератором — визуальный. Сейчас есть текстовый скрипт: список команд с программным счётчиком, регистрами `F0..F9`, `IF/ELSE/ENDIF`, `GOTO` и `DELAY`. Нужен экран, где тот же сценарий набирается нодами: нода задаёт параметры генератора и задержку, связи задают порядок.

Это дополнение, а не замена. Экран скриптов остаётся как есть.

## Принятые решения

| Решение | Выбор |
|---|---|
| Семантика связи | Секвенсор: один активный бегунок идёт по стрелкам, как по блок-схеме скрипта |
| Кто исполняет | Граф компилируется в строки скрипта, крутит существующий `Script` |
| Типы нод | Паритет с языком скрипта: Старт, Шаг, Регистр, Условие, Стоп |
| Диалог Шага | Полная форма всех параметров, галочка «менять» у каждого |
| Число или регистр | Чип `123` ⇄ `F` слева от поля значения |
| Жесты | Связь в два тапа; перетаскивание всегда означает «двигать ноду» |
| Прогон | Холст на весь экран, шторка снизу с регистрами и консолью, кнопка `{ }` показывает сгенерированный скрипт |

Отвергнутые варианты и почему — в конце документа.

### Что переиспользуется без изменений

`Script` целиком — цикл исполнения, `pc`, регистры, пауза, отмена, счётчик поколений, логгер. `parseCommand` и типы `GenBlock`, `GenParam`, `Operand`, `CompareOp` из `ScriptCommand.kt`: модель ноды строится прямо на них, параллельных перечислений не заводим. `Generator` граф не трогает — до генератора он дотягивается только через `Script`. `RegisterViewDraw` из экрана скриптов рисует регистры в шторке.

### Что меняется в существующем коде

- `AppPath` — папка `Nodes` (`/Gen3/Nodes`).
- `AppScreen` — экраны `Nodes` и `GeneratedScript`.
- `M4BottomAppBarComponent` — кнопка входа рядом с существующей `navigateToScript`.
- `ScriptCommand.kt` — из `parseCommand` наружу выносятся `parseOperand(token): Operand?` и `Operand.toToken(): String`; сейчас разбор операнда заперт в локальной функции. Обе стороны — файл `.ng` и язык скрипта — начинают использовать один код, и разъехаться уже не могут.
- `screens/scripting/dialog/DialogSaveAs.kt` и `DialogDeleteRename.kt` переезжают в `screens/common/dialog/` и вызываются с обоих экранов: они про имя файла, а не про скрипты.
- DI: `NodesModule`, `GeneratorArbiter`, `VMNodes` в `ScreenModelModule`.

### Границы

Компилятор и валидатор — чистые функции без Android: `NodeGraph -> CompileResult`. `NodeRunner` не знает про Compose, `VMNodes` не знает про генератор, холст не знает про компилятор — он рисует `NodeGraph` и подсвеченный `NodeId`.

### Владение генератором

`GeneratorArbiter` — синглтон, знает текущего владельца (`NONE`, `SCRIPT`, `NODES`). Запуск графа глушит текстовый скрипт, запуск скрипта глушит граф. Без этого два прогона пишут в `gen.liveData` одновременно и результат непредсказуем.

Чтобы не заводить круговую зависимость, арбитр не знает ни про `Script`, ни про `NodeRunner`: каждый владелец при старте зовёт `acquire(owner) { остановить себя }`, арбитр вызывает сохранённый обработчик прежнего владельца и запоминает нового.

`Script` — обычный класс, второй экземпляр создаётся свободно, поэтому у графа свой движок, и `list`/`name` открытого текстового скрипта он не затирает.

## Модель данных

```kotlin
@JvmInline value class NodeId(val value: Int)

data class NodeGraph(
    val nodes: List<GraphNode> = emptyList(),
    val edges: List<GraphEdge> = emptyList(),
)

data class GraphNode(
    val id: NodeId,
    val title: String,
    val x: Float,
    val y: Float,
    val body: NodeBody,
)

/** Выход ноды. У Старта, Шага и Регистра один, у Условия два, у Стопа ни одного. */
enum class Port { OUT, YES, NO }

data class GraphEdge(val from: NodeId, val port: Port, val to: NodeId)

enum class RegOp { LOAD, PLUS, MINUS }

sealed interface NodeBody {
    data object Start : NodeBody
    data object Stop : NodeBody
    data class Step(val params: StepParams, val delayMs: Long) : NodeBody
    data class Register(val op: RegOp, val dst: Int, val src: Operand) : NodeBody
    data class Condition(val left: Int, val op: CompareOp, val right: Operand) : NodeBody
}
```

`Operand` и `CompareOp` — те самые, из `ScriptCommand.kt`. Чип `123` ⇄ `F` в форме есть в точности выбор между `Operand.Const` и `Operand.Reg`.

### Параметры шага

22 поля, `null` означает снятую галочку.

```kotlin
data class StepParams(val ch1: ChannelParams, val ch2: ChannelParams)

data class ChannelParams(
    val carrierEnabled: Boolean? = null,   // CH1 CR ON|OFF
    val carrierFr: Operand?     = null,    // CR1 FR  <val>
    val carrierMod: String?     = null,    // CR1 MOD <файл>
    val amEnabled: Boolean?     = null,
    val amFr: Operand?          = null,
    val amMod: String?          = null,
    val fmEnabled: Boolean?     = null,
    val fmBase: Operand?        = null,    // FM1 BASE
    val fmDev: Operand?         = null,
    val fmFr: Operand?          = null,
    val fmMod: String?          = null,
)
```

`CR1 FR` и `FM1 BASE` пишут одно и то же поле `gen.liveData.ch1_Carrier_Fr` (`Script.kt`, `genValue`). Если отмечены оба, побеждает применённый вторым. Валидатор выдаёт на это предупреждение.

### Инварианты

- Один порт — одна связь. Новая связь из занятого порта заменяет старую без вопросов.
- `id` не переиспользуются: новый равен максимальному плюс один, чтобы связи не прилипали к другой ноде после удаления.
- Старт ровно один. Стопов сколько угодно, ноль тоже допустим — бесконечный цикл законен.
- Новый граф создаётся как `Старт → Стоп`: сразу валиден, можно жать «Пуск».

## Формат файла

`/Gen3/Nodes/<имя>.ng`, JSON через Gson (`kotlinx.serialization` в проекте отключён, Gson уже в зависимостях). Плоский DTO с дискриминатором `type`: sealed-классы и `value class` Gson не переваривает, а `RuntimeTypeAdapterFactory` здесь не нужен.

```json
{
  "version": 1,
  "nodes": [
    { "id": 1, "type": "START", "title": "Старт", "x": 40, "y": 120 },
    { "id": 2, "type": "STEP", "title": "Разгон", "x": 220, "y": 100,
      "delayMs": 100,
      "ch1": { "carrierEnabled": true, "carrierFr": "F1", "amMod": "02_HWave" },
      "ch2": {} },
    { "id": 3, "type": "REGISTER", "title": "шаг +50", "x": 220, "y": 240,
      "regOp": "PLUS", "regDst": 1, "regSrc": "50" },
    { "id": 4, "type": "CONDITION", "title": "до 5 кГц", "x": 220, "y": 350,
      "condLeft": 1, "condOp": "<", "condRight": "5000" },
    { "id": 5, "type": "STOP", "title": "Стоп", "x": 40, "y": 460 }
  ],
  "edges": [
    { "from": 1, "port": "OUT", "to": 2 },
    { "from": 2, "port": "OUT", "to": 3 },
    { "from": 3, "port": "OUT", "to": 4 },
    { "from": 4, "port": "YES", "to": 2 },
    { "from": 4, "port": "NO",  "to": 5 }
  ]
}
```

Операнд хранится строкой — `"1000.0"` или `"F1"`, тем же токеном, что уйдёт в скрипт. Ни адаптеров Gson, ни второго представления числа, файл читается глазами, отсутствующий ключ означает снятую галочку.

`NodeGraphUtils` повторяет API `ScriptUtils`: список имён, сохранить, прочитать, удалить, переименовать.

## Компилятор

```kotlin
sealed interface CompileResult {
    data class Ok(
        val lines: List<String>,
        val lineToNode: List<NodeId>,   // индексы совпадают с lines
        val warnings: List<Issue>,
    ) : CompileResult

    data class Failed(val errors: List<Issue>, val warnings: List<Issue>) : CompileResult
}

data class Issue(val nodeId: NodeId?, val severity: Severity, val text: String)
enum class Severity { ERROR, WARNING }
```

Обход в глубину от преемника Старта, преемники в порядке портов `OUT`, `YES`, `NO`. Порядок влияет только на читаемость: все переходы — явные `GOTO`. Зато он детерминирован, и golden-тесты стабильны.

Два прохода: сначала размер блока каждой ноды и её адрес, затем строки с уже известными номерами.

| Нода | Строк | Что печатает |
|---|---|---|
| Старт | 0 | точка входа: `pc = 0` есть первая строка его преемника |
| Шаг | *n* + (задержка > 0) + 1 | значения, затем переключатели, затем `DELAY`, затем `GOTO` |
| Регистр | 2 | `PLUS F1 50.0`, `GOTO …` |
| Условие | 5 | `IF F1 < 5000`, `GOTO <да>`, `ELSE`, `GOTO <нет>`, `ENDIF` |
| Стоп | 1 | `END` |

Пример Шага:

```
CR1 MOD 02_HWave
CR1 FR F1
CH1 CR ON
DELAY 100
GOTO 7
```

Внутри Шага порядок канонический: сначала значения (`MOD`, `FR`, `BASE`, `DEV`), потом переключатели (`ON`/`OFF`), сначала канал 1, потом канал 2. Значения доезжают до генератора раньше, чем блок включается, — блок оживает уже на нужной частоте.

У Условия ровно пять строк, потому что движок на истинном условии ставит `pc = current + 1` — это `GOTO <да>`, а на ложном зовёт `findPairLine(stopOnElse = true)`, который доходит до `ELSE` и возвращает `i + 1` — это `GOTO <нет>`. `ENDIF` не исполняется никогда, но обязан существовать, иначе `findPairLine` бросит «не найден парный ENDIF».

Загрузка в движок: `script.list.clear()`, затем `add(line)` на каждую строку. API `ScriptList` уже подходит.

### Самопроверка

Строки печатает компилятор, поэтому любая ошибка разбора в рантайме — его баг. Перед возвратом `Ok`:

1. каждая строка прогоняется через `parseCommand`;
2. каждый `GOTO` проверяется на попадание в диапазон;
3. `lines.size == lineToNode.size`.

Не сошлось — `Failed` с «внутренняя ошибка компилятора».

### Подсветка активной ноды

```kotlin
val activeNode: StateFlow<NodeId?> = script.pc.map { lineToNode.getOrNull(it) }
```

`script.pc` уже `MutableStateFlow<Int>`. На `DELAY` движок держит `pc` на самой строке задержки, а она принадлежит своему Шагу, — значит нода светится ровно столько, сколько длится её пауза.

## Валидатор

Ошибки, «Пуск» неактивен:

1. Старта нет или их больше одного.
2. У Старта нет исходящей связи.
3. У Шага или Регистра нет исходящей связи.
4. У Условия пустой выход «да» или «нет».
5. Нода недостижима от Старта.
6. Связь ведёт в несуществующую ноду.
7. Отрицательная задержка либо номер регистра вне `F0..F9`.

Правила 6 и 7 срабатывают только на файле, который правили руками: интерфейс такого не даёт.

Предупреждения, пуск разрешён:

8. Цикл, в котором сумма задержек равна нулю. Устройство не повиснет — в `Script` есть `YIELD_EVERY = 256`, — но ядро будет греться. Обнаружение: сильно связные компоненты по Тарьяну, в каждой нетривиальной компоненте суммируем задержки Шагов; `O(V+E)`, без перебора циклов.
9. Шаг без отмеченных полей и с нулевой задержкой — ничего не делает.
10. В одном Шаге отмечены и `CR FR`, и `FM BASE` — пишут одно поле.
11. Имя формы сигнала (`MOD`) неизвестно генератору.

### Откуда берутся имена форм

`Spinner_Send_Buffer` ищет имя не в папке, а в уже загруженных в память списках: несущая — в `gen.itemlistCarrier`, **и AM, и FM — в `gen.itemlistAM`** (`Spinner_Send_Buffer.kt`). Неизвестное имя молча игнорируется: `index == -1` и выход, форма остаётся прежней.

Чтобы валидатор остался чистой функцией, имена приходят параметром:

```kotlin
fun validate(
    graph: NodeGraph,
    carrierNames: Set<String>,
    modNames: Set<String>,
): List<Issue>
```

`VMNodes` передаёт сюда имена из `gen.itemlistCarrier` и `gen.itemlistAM`, тесты — фикстуру либо пустые множества.

Те же списки питают выпадающие списки формы в диалоге Шага: поле `MOD` выбирается из известных имён, а не набирается руками. Так предупреждение 11 останется только для графов, принесённых с другого устройства.

## Экран

### Холст

Ноды — обычные composable, разложенные по мировым координатам внутри одного слоя с `graphicsLayer(scaleX/scaleY = scale, translationX/Y = offset)`. Рёбра рисуются `Canvas`-ом в том же слое, под карточками. Compose учитывает трансформации слоя при доставке касаний, поэтому hit-testing нод достаётся даром.

Это единственное место в дизайне, где мы полагаемся на поведение фреймворка, а не на свой код. Проверять его надо первым, до остальной работы по холсту. Запасной план, если попадания «поедут»: рисовать всё одним `Canvas` и делать hit-test вручную обратной трансформацией.

Карточка фиксированного размера, 168 × 72 dp. Тогда якоря рёбер считаются арифметикой — без `onSizeChanged`, без карты размеров и без дёрганья рёбер на первом кадре. Содержимое, если не влезло, обрезается многоточием.

Жесты: перетаскивание карточки двигает ноду (мировые координаты увеличиваются на `pan / scale`), перетаскивание по пустому месту панорамирует, щипок масштабирует в пределах `0.4…2.5`. Кнопка «вписать граф» считает габариты и подгоняет `scale` с `offset`.

### Режим связи

В `VMNodes` — `linkFrom: (NodeId, Port)?`. Пока он не `null`: холст притемнён, карточки-кандидаты подсвечены, тап по карточке создаёт ребро, тап по пустому месту отменяет.

Цель — любая нода, кроме Старта: в него нельзя войти. Петля на себя разрешена — `Шаг` на самого себя с задержкой есть законное «крутись, пока не нажму стоп».

Тапа по кривой нет. Правило «один порт — одна связь» означает, что связь однозначно задаётся парой `(нода, порт)`, поэтому создаётся, переподключается и удаляется она из панели ноды-источника: у занятого порта кнопка становится «Переподключить» и «Отвязать». Это убирает hit-test кривых Безье, ничего не отнимая.

### Панель действий

| Выделено | Кнопки |
|---|---|
| Шаг, Регистр | Параметры, Связь, Дублировать, Удалить |
| Условие | Параметры, Связь «да», Связь «нет», Дублировать, Удалить |
| Старт | Связь |
| Стоп | Дублировать, Удалить |

Ничего не выделено — панели нет, холст на весь экран.

«Дублировать» создаёт копию тела ноды **без связей**, со смещением на 24 dp вправо и вниз, и сразу выделяет её. «Удалить» убирает ноду вместе со всеми входящими и исходящими рёбрами.

### Диалог Шага

Группы `CH1 · Несущая`, `CH1 · AM`, `CH1 · FM`, `CH2 · …` свёрнуты, в заголовке группы счётчик отмеченных полей вида «2 из 3». Отмеченные строки подсвечены, неотмеченные приглушены. Каждое числовое поле имеет слева чип `123` ⇄ `F`: в режиме `123` обычный ввод, в режиме `F` выпадающий список `F0..F9`. Поля `MOD` — выпадающие списки известных генератору форм. Внизу поле задержки и кнопка «Снять с генератора», которая заполняет форму текущим состоянием генератора и ставит галочки на все поля.

### Когда что считается

Валидация — на каждое изменение графа: он маленький, а проверка `O(V+E)`, так что состояние кнопки «Пуск» и бейдж с числом проблем всегда актуальны. Компиляция — только по «Пуску» и при открытии `{ }`.

### Прогон

Состояния берутся из `script.state` (`StateCommandScript`), своих не заводим; графу нужны три — `ISRUNNING`, `ISPAUSE`, `ISTOPPING`. Отдельного режима правки, как на экране скриптов, здесь нет: пока прогон не остановлен, «Параметры», «+», «Дублировать» и «Удалить» сначала предлагают остановиться.

Шторка снизу: свёрнутая — строка с занятыми регистрами и «консоль ▴»; развёрнутая — вкладки «Регистры» и «Консоль».

Консоль своя, не общая: `consoleLog` в `ScriptLog.kt` — глобальная `var` верхнего уровня, и запись в неё смешала бы лог графа с логом скрипта. `VMNodes` заводит собственный `Console2()` и подставляет его в `script.logger` своего экземпляра движка.

Кнопка `{ }` открывает `GeneratedScriptScreen` — текст скомпилированного скрипта только для чтения, с подсветкой текущей строки. Это основной инструмент отладки самого компилятора. Если граф не проходит валидацию, экран показывает вместо текста список ошибок.

### Файлы и навигация

Меню в топбаре: Новый, Открыть, Сохранить, Сохранить как, Переименовать, Удалить. Диалоги «Сохранить как» и «Переименовать/Удалить» переиспользуются из `screens/common/dialog/`.

Флаг `dirty` взводится на любое изменение графа, включая перемещение ноды: координаты хранятся в файле. «Новый» и «Открыть» при взведённом `dirty` спрашивают то же самое, что и уход с экрана.

Масштаб и смещение холста в файле не хранятся: при открытии графа холст сам вписывает его в экран.

Навигация: `AppScreen.Nodes` и `AppScreen.GeneratedScript`, вход — кнопка в `M4BottomAppBarComponent`.

## Ошибки

| Что | Реакция |
|---|---|
| Валидация не прошла | «Пуск» неактивен, рядом бейдж с числом проблем. Тап открывает список; тап по строке центрирует холст на виноватой ноде и выделяет её. Предупреждения в том же списке, жёлтым, пуск не блокируют. |
| Битый `.ng`: не тот JSON, неизвестный `type`, `version` новее нашей | Снекбар с ошибкой, файл не открывается, граф в памяти не трогаем — неудачное открытие не должно стирать работу. |
| Файл не записался: нет прав, нет места | Снекбар с ошибкой, флаг `dirty` остаётся взведён. |
| Внутренняя ошибка компилятора | Снекбар и строка в консоль графа, прогон не начинается. |
| `ScriptException` в рантайме | Случиться не должна. В консоль «Внутренняя ошибка в ноде "X"» — нода берётся из `pc` через `lineToNode`, движок оставляет `pc` на битой строке. Движок сам уходит в `ISTOPPING`. |
| Уход с экрана с несохранённым графом | Диалог «Сохранить», «Не сохранять», «Отмена». |

Снекбары идут через существующий `EventBus` и фасад `SnackBar`.

## Тестирование

`Generator` — обычный класс без Android-зависимостей, `Script(gen)` берёт только его. Значит настоящий движок запускается в JVM-тесте, и проверять можно не «строки похожи на ожидаемые», а «граф действительно крутит генератор». Понадобится добавить `testImplementation` на `kotlinx-coroutines-test`: сейчас в тестах только JUnit 4.13.2.

1. **Сквозной тест, главный.** Граф свипа: `Старт → Шаг(CR1 FR F1, задержка 100) → Регистр(F1 += 50) → Условие(F1 < 1200)`, «да» ведёт в Шаг, «нет» в Стоп. Компилируем, грузим в живой `Script(Generator())`, гоняем на `TestScope` с виртуальным временем, проверяем, что `gen.liveData.ch1_Carrier_Fr` прошла 1000, 1050, …, 1200 и движок встал. Разом проверяет компилятор, адреса переходов, ветвление Условия и семантику задержки.
2. **Golden-тесты компилятора**: точные строки и номера для линейной цепочки и для цикла со свипом. Ловят непреднамеренные сдвиги адресов.
3. **Инварианты компилятора** на наборе графов: каждая строка парсится `parseCommand`, каждый `GOTO` в диапазоне, `lines.size == lineToNode.size`, каждая нода кроме Старта упомянута в `lineToNode`.
4. **Валидатор**: по тесту на каждое из семи правил-ошибок и четырёх предупреждений. Для цикла без задержки — и положительный, и отрицательный случай.
5. **Round-trip формата**: граф → JSON → граф равен исходному; отсутствующий ключ даёт `null`; неизвестный `type` даёт ошибку чтения; `parseOperand` и `toToken` в обе стороны.

Имена тестов — в бэктиках по-русски, как в `ScriptCommandTest`.

Интерфейс автоматически не тестируем: инструментальных тестов в проекте нет, разворачивать стенд ради одного экрана несоразмерно. Холст, жесты и диалоги проверяются руками.

## Чего в первой версии не будет

Отмена и повтор действий, выделение нескольких нод, копирование нод между графами, макро-ноды «Свип» и «Повторить N раз», импорт существующих `.sk` в граф, автораскладка.

Макро-ноды отложены сознательно: каждая разворачивается в цикл со скрытым регистром, а значит нужен распределитель свободных регистров и ошибка «регистры кончились». Поверх готовой первой версии это добавляется, ничего не ломая.

## Отвергнутые варианты

**Node-RED с параллельными ветками** — нода с несколькими выходами размножала бы бегунка, и каналы шли бы независимыми цепочками со своими задержками. Отвергнут: нужны правила остановки и слияния веток, отладка заметно сложнее, а задача была в визуальном аналоге скрипта.

**Патч сигналов в духе PureData** — ноды как звуковые блоки, провода как маршрут сигнала, граф работает непрерывно. Отвергнут: это переделка аудиодвижка, а не аналог скрипта, и «задержка в ноде» в такую модель не ложится.

**Свой исполнитель графа вместо компиляции** — `NodeRunner` ходил бы по нодам напрямую. Отвергнут: заново писать цикл, паузу, отмену и счётчик поколений — самое коварное место существующего движка, и дальше чинить баги пришлось бы в двух местах.

**Граф и текст как два вида одного документа** — правки в любом видны в другом. Отвергнут: обратный разбор произвольного текста в граф есть отдельный тяжёлый проект, правка текста ломает раскладку, ошибка любой стороны бьёт по обеим.

**Список присваиваний вместо формы** в диалоге Шага — строки «блок, параметр, значение», как в скрипте. Отвергнут в пользу формы: форма показывает сразу всё и не требует выбирать параметр из списков.

**Тянуть провод от порта к порту** — как в node-red. Отвергнут: на телефоне порт должен быть не меньше 44 dp, иначе в него не попасть, а такие «уши» съедают карточку; при отдалении вместо провода тянется нода.

**Автораскладка без свободного размещения** — координаты считает алгоритм. Отвергнут: при циклах и ветвлениях автораскладка почти всегда рисует не то, что имелось в виду, а поправить нельзя.
